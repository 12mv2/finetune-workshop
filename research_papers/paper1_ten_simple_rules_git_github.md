# Ten Simple Rules for Taking Advantage of Git and GitHub

**Published:** July 14, 2016
**DOI:** https://doi.org/10.1371/journal.pcbi.1004947
**Journal:** PLOS Computational Biology

## Authors
Yasset Perez-Riverol, Laurent Gatto, Rui Wang, Timo Sachsenberg, Julian Uszkoreit, Felipe da Veiga Leprevost, Christian Fufezan, Tobias Ternent, Stephen J. Eglen, Daniel S. Katz, Tom J. Pollard, Alexander Konovalov, Robert M. Flight, Kai Blin, Juan Antonio Vizcaíno

## Overview

This editorial piece provides practical guidance for researchers on leveraging Git and GitHub for scientific software development and collaborative projects. The authors emphasize that version control and open-source code sharing are essential for reproducible science.

## Key Rules

**Rule 1: Use GitHub to Track Your Projects**
The paper recommends initializing local repositories and committing changes frequently. Users can synchronize local and remote repositories using basic commands like `git push` and `git pull`. The distributed nature of Git provides fault tolerance through multiple backup copies.

**Rule 2: GitHub for Single Users, Teams, and Organizations**
GitHub enables permission management at multiple levels. Researchers can grant collaborators write access, while organizations facilitate team-based project management across institutes and large open-source initiatives.

**Rule 3: Developing and Collaborating on New Features: Branching and Forking**
Forking creates independent copies for experimentation. Feature branches allow parallel development, and pull requests enable peer review before merging changes into the main codebase.

**Rule 4: Naming Branches and Commits: Tags and Semantic Versions**
Semantic versioning (X.Y.Z format) provides clear documentation of changes. Tags enable researchers to recover specific versions and reproduce published results.

**Rule 5: Let GitHub Do Some Tasks for You: Integrate**
Continuous integration through platforms like Travis CI automatically tests code after each push, ensuring quality and preventing errors from being merged into main branches.

**Rule 6: Let GitHub Do More Tasks for You: Automate**
Services like Codecov measure test coverage, while Read the Docs auto-generates documentation. These integrations create quality badges demonstrating project maintenance.

**Rule 7: Use GitHub to Openly and Collaboratively Discuss, Address, and Close Issues**
GitHub's issue tracker supports collaboration beyond bug reporting. Labels, milestones, and assignees help prioritize tasks. Pull request templates standardize contributions.

**Rule 8: Make Your Code Easily Citable, and Cite Source Code!**
Integration with Zenodo and Figshare enables Digital Object Identifiers (DOIs) for code repositories. This practice supports reproducibility and gives developers proper academic credit.

**Rule 9: Promote and Discuss Your Projects: Web Page and More**
GitHub Pages host project documentation. Real-time communication platforms like Gitter facilitate community support, while wikis and Gists share code snippets and resources.

**Rule 10: Use GitHub to Be Social: Follow and Watch**
Following users and watching projects helps researchers stay current with relevant developments in their field.

## Supporting Files
The paper includes supplementary materials on Git Large File Storage, testing practices, and source code documentation.

## Conclusion

The authors encourage researchers to embrace Git and GitHub despite the initial learning curve, highlighting examples of successful bioinformatics projects using these tools for collaborative, transparent, and reproducible research.
